# 油气工程实验教学与安全考核平台详细设计说明

> 本说明依据当前仓库中的前端路由与页面、后端控制器/服务/实体以及主数据库脚本整理。文档描述的是 2026-07-11 工作区现状，包括尚未提交但已存在于工作区的考试回看、主观题批改和分步骤学习记录设计。

## 1. 文档目的与设计范围

本文档用于说明系统各业务模块的建设目的、主要需求、实现逻辑和数据依赖，为开发、测试、验收、部署和后续维护提供统一依据。

系统面向游客/普通用户、学生、教师和管理员，形成“资源学习—课堂教学—安全考核—实验准入—实验预约—报告评价”的实验教学闭环，同时提供门户消息、讨论交流、智能推荐、AI 问答和后台治理能力。

### 1.1 当前技术基线

- 前端采用 Vue 3、Vue Router、Pinia、Element Plus 和 Vite，路由元数据保存页面所需角色或权限码。
- 后端采用 Spring Boot、Java、MyBatis-Plus，按 Controller、Service、Mapper、POJO/DTO/VO 分层。
- 前后端通过 `/api/**` REST 接口通信，统一使用 Token 认证和业务结果对象返回数据。
- 主数据库为 MySQL，主 SQL 脚本当前定义 43 张业务表。
- 上传文件保存在后端运行目录的 `uploads` 目录，通过受控接口或 `/uploads/**` 静态映射访问。

### 1.2 总体业务关系

```mermaid

    U["用户与权限"] --> P["门户与消息"]
    U --> C["课堂管理"]
    C --> E["实验与学习路径"]
    E --> R["教学资源学习"]
    E --> X["安全考核"]
    X --> A["实验准入"]
    A --> B["实验预约"]
    B --> G["实验报告与评价"]
    C --> D["讨论交流"]
    R --> I["推荐与 AI 辅助"]
    C --> S["统计看板"]
    X --> S
    B --> S

```

主业务链要求前序条件逐步满足：用户必须先取得课堂与资源访问权限，完成必要学习后参加安全考试；考试通过形成实验准入，准入有效时才可预约实验；实验完成后提交报告并由教师按评分规则评价。

## 2. 用户认证、账户与权限模块

### 2.1 模块说明

- **基本描述**：统一管理注册、登录、退出、密码修改、个人资料、用户、角色、权限及教师身份认证。
- **对应需求**：保证所有受保护接口只对已登录用户开放，并依据用户角色和权限码限制不同页面、按钮与业务操作。
- **建设目的与意义**：把身份、角色和业务授权从具体业务模块中分离，避免学生越权执行教师操作、普通用户进入管理员功能或用户访问他人私有数据。
- **主要建设内容**：账号密码校验、Token 签发与失效、个人信息维护、用户增删改查、角色权限配置、教师资格申请与审核。
- **模块特点**：采用 RBAC 模型；前端路由守卫负责页面入口控制，后端 `LoginInterceptor` 负责认证，`PermissionAspect` 对 `@RequirePermission` 接口执行最终授权。
- **核心实现**：`AuthController`、`UserController`、`RoleController`、`PermissionController`、`TeacherCertificationController`、`LoginInterceptor`、`PermissionAspect`。

### 2.2 登录与接口鉴权流程

```mermaid
sequenceDiagram
    actor User as 用户
    participant FE as Vue 前端
    participant Auth as AuthController
    participant DB as 用户与令牌表
    participant Interceptor as LoginInterceptor
    participant AOP as PermissionAspect
    participant API as 业务接口

    User->>FE: 输入账号和密码
    FE->>Auth: POST /api/auth/login
    Auth->>DB: 查询用户并校验密码、状态
    alt 账号有效
        Auth->>DB: 创建或更新 Token
        Auth-->>FE: Token、用户信息、角色和权限
        FE->>FE: 保存会话并进入允许的首页
    else 账号无效或被禁用
        Auth-->>FE: 返回明确业务错误
        FE-->>User: 展示错误信息
    end
    User->>FE: 发起受保护操作
    FE->>Interceptor: 请求头携带 Authorization
    Interceptor->>DB: 校验 Token 及有效期，加载角色权限
    Interceptor->>AOP: 写入 UserContext 后继续
    AOP->>AOP: 检查 @RequirePermission
    alt 具有权限
        AOP->>API: 执行业务方法
        API-->>FE: 返回业务结果
    else 权限不足
        AOP-->>FE: 返回 403 类业务错误
    end
```

文字说明：登录和注册是 `/api/**` 下仅有的免登录入口。其他请求必须携带有效 Token；拦截器从令牌反查用户的角色和权限并保存到当前请求上下文。页面权限只改善交互体验，后端权限切面才是安全边界。请求结束后清理线程上下文，避免复用线程时发生用户信息串用。

## 3. 公共门户、公告与消息日程模块

### 3.1 模块说明

- **基本描述**：向不同身份用户提供统一首页、公告、站内消息、未读数、日历事件、全局搜索、最近访问和快捷入口。
- **对应需求**：用户登录后能够及时看到待办事项、考试/预约/报告提醒和平台公告，并快速返回高频业务页面。
- **建设目的与意义**：将分散在考试、预约、报告等模块的信息聚合，降低用户寻找任务的成本，减少遗漏关键时间节点。
- **主要建设内容**：角色化首页数据组装、公告发布与上下线、消息已读管理、日历聚合、跨模块搜索、访问历史和个人快捷方式配置。
- **模块特点**：门户数据按用户身份动态组装；消息和日历是业务模块的统一出口；管理员可管理公告并手工触发提醒任务。
- **核心实现**：`PortalController`、`AdminPortalController`、`PortalService`、`PortalMessageService`、`PortalReminderService`。

### 3.2 门户聚合流程


文字说明：门户不复制各业务数据，而是从相关服务读取当前用户可见的信息后组装视图对象。提醒服务根据临近的考试、预约或报告节点生成站内消息；用户阅读后只更新个人消息的已读状态，不影响公告本身。

## 4. 课堂、教学班与成员管理模块

### 4.1 模块说明

- **基本描述**：教师创建和维护课堂，将实验内容、教学班和学生组织到具体教学空间中；学生可通过导入名单或邀请码加入课堂。
- **对应需求**：支持教师按课程开展分班教学，控制课程发布、归档和成员范围，并为后续学习、考试和统计提供课程上下文。
- **建设目的与意义**：把公共资源与正式教学活动区分开，使实验、任务、考试、讨论和成绩能够按课堂隔离。
- **主要建设内容**：课堂增删改查、发布/归档、课堂复制、教学班维护、学生导入与移除、邀请码生成/停用和学生加入课堂。
- **模块特点**：课堂是教学域聚合根；教师对自己创建或负责的课堂执行管理操作；邀请码可以控制有效期、使用次数和启停状态。
- **核心实现**：`CourseController`、`CourseService`、`ClassInviteService`、`CourseList.vue`、`CourseManagement.vue`、`CourseEditor.vue`。

### 4.2 课堂建设与加入流程


## 5. 实验内容、步骤与学习任务模块

### 5.1 模块说明

- **基本描述**：教师维护实验目标、风险、安全要求、准入条件和分步骤操作内容，学生按照课堂学习路径完成章节、步骤和任务。
- **对应需求**：把实验前知识学习、风险辨识和操作准备结构化，能够追踪学生学习进度，而不是仅提供文件下载。
- **建设目的与意义**：通过可检查的前置学习和任务确认提高实验安全性，为考试准入和教师统计提供过程证据。
- **主要建设内容**：实验增删改查、状态维护、实验步骤编辑、学习任务配置、检查清单确认、任务完成分布和学习进度记录。
- **模块特点**：实验由有序步骤组成；任务可形成路径节点；当前工作区新增 `t_step_learning_record`，用于精确记录每个实验步骤是否完成。
- **核心实现**：`ExperimentController`、`LearningTaskController`、`LearningRecordController`、`ExperimentService`、`LearningTaskService`、`StepLearningRecordService`。

### 5.2 学习路径活动图


文字说明：页面先读取实验概览和学习路径，再按步骤展示文字、资源和安全提示。完成步骤时接口使用“用户 + 步骤”定位记录，避免重复写入；总体进度由步骤和任务完成情况计算或更新。检查清单用于确认学生已理解关键注意事项，但不能替代安全考试准入。

## 6. 教学资源学习、互动与投稿模块

### 6.1 模块说明

- **基本描述**：统一管理视频、文档、图片、链接等教学资源，支持检索、预览、学习、互动、时间轴笔记、统计和用户投稿审核。
- **对应需求**：学生既能从课堂实验进入定向资源，也能在公共资源中心自主学习；教师能够建设资源，管理员能够审核公开投稿。
- **建设目的与意义**：沉淀可复用的油气工程实验知识与安全教育内容，并通过学习行为数据发现资源质量和教学难点。
- **主要建设内容**：资源上传与元数据维护、状态控制、查看/下载统计、评分收藏等互动、视频时间轴笔记、实验热点统计、投稿及审核、公开资源预览。
- **模块特点**：资源可关联课程、实验、知识点和风险类型；文件访问经过资源权限判断；时间轴笔记适用于视频定位和问题追踪。
- **核心实现**：`ResourceController`、`FileController`、`ResourceSubmissionController`、`ResourceService`、`ResourceTimelineNoteService`。

### 6.2 资源发布与学习流程

文字说明：资源元数据与物理文件分开管理。预览和下载前需检查资源状态及当前用户的可见范围；查看、下载和互动分别形成统计数据。用户投稿先进入独立审核表，只有审核通过后才能作为公共资源对外展示。

## 7. 题库、试卷与安全考试模块

### 7.1 模块说明

- **基本描述**：教师维护安全题库，按课程和实验创建试卷、配置题目及考试规则；学生在线答题，系统保存答案、客观题判分并支持主观题人工批改和考试回看。
- **对应需求**：通过安全知识考核验证学生是否理解实验风险与操作规范，并把考试结果作为实验准入的重要依据。
- **建设目的与意义**：让安全培训从“已浏览”转为“可验证掌握”，保留完整答题证据，支持错题复习和教学分析。
- **主要建设内容**：题库增删改查与批量创建、试卷创建/编辑/删除/发布、题目分值和顺序、智能组卷、考试开始/暂存/提交、自动判分、主观题批改、记录回看、错题与统计分析。

- **核心实现**：`QuestionController`、`ExamPaperController`、`ExamController`、`ExamStatisticsController`、`QuestionService`、`ExamPaperService`、`ExamService`。

## 8. 实验准入与预约模块

### 8.1 模块说明

- **基本描述**：根据安全考试结果形成实验准入资格；教师维护实验室开放时段和容量，学生在准入有效的前提下提交预约，教师审核预约。
- **对应需求**：保证只有完成安全要求的学生才能预约实验，并对实验室时段容量和申请状态进行统一控制。
- **建设目的与意义**：把考试结果和线下实验安排连接起来，降低无资格预约、超容量预约和时间冲突风险。
- **主要建设内容**：准入状态查询、实验时段维护、可预约时段查询、预约提交、我的预约、取消预约、待审核列表和审核处理。
- **模块特点**：预约创建需同时校验身份、准入、时段状态、容量、重复预约和时间条件；审核和取消会影响已预约数量，必须保持状态与容量一致。
- **核心实现**：`AdmissionService`、`ReservationController`、`ReservationService`、`LabTimeSlot`、`Reservation`。

- **主要建设内容**：用户管理、角色授权、权限树展示、公告治理、教师认证审核、资源投稿审核、操作日志检索、提醒任务执行。
- **模块特点**：管理接口使用细粒度权限码；审核必须记录操作者、时间和意见；日志以查询为主，不提供普通用户删除入口。
- **核心实现**：`UserController`、`RoleController`、`PermissionController`、`AdminPortalController`、`TeacherCertificationController`、`ResourceSubmissionController`。

## 15. 跨模块关键约束

### 15.1 权限与数据范围

- 所有 `/api/**` 请求除登录和注册外均需有效 Token。
- 页面路由权限不能替代后端接口权限；所有变更类接口必须以后端校验为准。
- 教师权限除权限码外还需校验课程/实验所有权或管理范围。
- 学生只能读取和修改自己的学习、考试、预约和报告记录。
- 管理员审核操作必须保留审核人、审核时间和审核意见。

### 15.2 状态一致性

- 课程、资源、试卷未发布时，不应出现在学生正式入口。
- 考试成绩只有完成全部必要判分后才可最终决定通过与准入。
- 预约通过、拒绝或取消时，应同步维护时段容量占用，避免负数或超额。
- 报告只有提交后才能进入教师待评列表，退回后允许学生修改并再次提交。
- 审核类记录应采用明确状态迁移，避免通过直接覆盖记录丢失历史责任信息。

### 15.3 事务边界

- 试卷提交：答案落库、判分、考试记录汇总和准入更新应处于可靠事务边界。
- 预约审核/取消：预约状态与时段容量更新应原子完成。
- 报告量规评分：总评分与逐项评分应一致提交，汇总分由明细计算或严格校验。
- 角色授权：旧角色权限关系替换与新关系写入应整体成功或整体回滚。
